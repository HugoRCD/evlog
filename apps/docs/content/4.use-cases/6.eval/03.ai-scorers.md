AI scorers evaluate the **behavior** of the model, not just the output. They read `ctx.event.ai.*` — the telemetry fields that `createAILogger` captures automatically for every case.

This is what separates `@evlog/eval` from other eval frameworks: because token usage, tool calls, cache hits, latency, and finish reason are already on the wide event, you can assert on them with the same scorer API.

## Import

```typescript
import {
  maxSteps, maxCalls,
  noToolLoop, maxToolCallFrequency,
  minCacheHitRate,
  finishesCleanly,
  maxCost, maxLatency,
  maxTokensPerStep, minThroughput,
} from '@evlog/eval/ai-scorers'
```

## Reference

| Scorer | Passes when | Field |
|--------|-------------|-------|
| `maxSteps(n)` | `ai.steps <= n` | `ai.steps` |
| `maxCalls(n)` | `ai.calls <= n` | `ai.calls` |
| `noToolLoop(threshold?)` | no tool called ≥ threshold times | `ai.toolCalls` |
| `maxToolCallFrequency(tool, max)` | specific tool called ≤ max times | `ai.toolCalls` |
| `minCacheHitRate(rate)` | `cacheReadTokens / inputTokens >= rate` | `ai.cacheReadTokens`, `ai.inputTokens` |
| `finishesCleanly` | `ai.finishReason === 'stop'` | `ai.finishReason` |
| `maxCost(dollars)` | `ai.estimatedCost <= dollars` | `ai.estimatedCost` |
| `maxLatency(ms)` | `ai.totalDurationMs <= ms` | `ai.totalDurationMs` |
| `maxTokensPerStep(n)` | `totalTokens / steps <= n` | `ai.totalTokens`, `ai.steps` |
| `minThroughput(t/s)` | `ai.tokensPerSecond >= t/s` | `ai.tokensPerSecond` |

## Detecting Agent Inefficiency

In May 2026, the GitHub team published a deep-dive on token efficiency in their CI agentic workflows. They found:

- One workflow had `search_repositories` called **342 times** in a single run (58% of all tool calls) — it should have been called 0 times.
- An agent that normally completes in 4 LLM turns was taking 18 turns after a misconfiguration.
- A bash allowlist bug caused a 64-turn fallback loop where the agent manually re-read source files the compiler should have read.

These patterns are invisible in output-based evals. With AI scorers, you can assert on them directly:

```typescript
import { createEval } from '@evlog/eval'
import { maxSteps, maxCalls, noToolLoop, maxToolCallFrequency, finishesCleanly, maxCost } from '@evlog/eval/ai-scorers'
import { generateText, tool } from 'ai'
import { z } from 'zod'

const agentEval = createEval({
  name: 'triage-agent-efficiency',
  dataset: [
    { id: 'pr-small', input: { diff: '10 lines changed', labels: [] } },
    { id: 'pr-large', input: { diff: '200 lines changed', labels: ['security'] } },
  ],
  task: async (input, { ai }) => {
    const model = ai.wrap('anthropic/claude-haiku-4')
    const result = await generateText({
      model,
      tools: {
        searchRepos: tool({ parameters: z.object({ q: z.string() }), execute: async () => ({ repos: [] }) }),
        addLabel:    tool({ parameters: z.object({ label: z.string() }), execute: async () => ({ ok: true }) }),
      },
      maxSteps: 15,
      prompt: `Triage this PR: ${JSON.stringify(input)}`,
    })
    return result.text
  },
  scorers: [
    maxSteps(8),                              // alert if the agent uses more than 8 turns
    maxCalls(8),
    noToolLoop(5),                            // fail if any tool is called 5+ times
    maxToolCallFrequency('searchRepos', 2),   // searchRepos should be called at most twice
    finishesCleanly,                          // finishReason must be 'stop'
    maxCost(0.005),                           // max $0.005 per case
  ],
  trialCount: 3,
  threshold: 0.85,
})

const results = await agentEval.run()
// results.cases[0].scores → { 'max-steps(8)': true, 'no-tool-loop(5)': false, ... }
// results.cases[0].ai    → { calls: 3, steps: 3, toolCalls: ['searchRepos', ...] }
```

## Step Count

```typescript
maxSteps(8)   // fail if ai.steps > 8
maxCalls(10)  // fail if ai.calls > 10
```

`steps` is the number of times the model was called in a multi-step agent loop. `calls` is the total number of LLM API calls (can differ when one step makes multiple calls).

Use both together to detect loops: constant `calls` with rising `steps` means the agent is looping without making progress.

## Tool Call Frequency

```typescript
// Any tool called 10+ times in a run fails (default threshold)
noToolLoop()

// Stricter: any tool called 5+ times fails
noToolLoop(5)

// Specific tool: searchRepos called more than 2 times fails
maxToolCallFrequency('searchRepos', 2)
```

Both scorers return `metadata` with the observed frequency, making it easy to inspect in drain output:

```json
{
  "scores": { "no-tool-loop(5)": false },
  "scoreMetadata": {
    "no-tool-loop(5)": {
      "maxFreq": 12,
      "loopingTool": "searchRepos",
      "threshold": 5
    }
  }
}
```

## Cache Efficiency

Prompt cache reads cost 10× less than fresh input tokens across all major providers. Low cache hit rates waste money.

```typescript
minCacheHitRate(0.3) // at least 30% of input tokens should come from cache
```

The scorer also attaches metadata:

```json
{
  "scores": { "min-cache-hit-rate(0.3)": false },
  "scoreMetadata": {
    "min-cache-hit-rate(0.3)": {
      "hitRate": 0.12,
      "cacheReadTokens": 540,
      "inputTokens": 4500,
      "target": 0.3
    }
  }
}
```

## Finish Reason

```typescript
finishesCleanly // passes only when ai.finishReason === 'stop'
```

Fails for `'error'`, `'max-tokens'`, or `'tool-calls'` — all signs the agent did not complete cleanly. Use this alongside `maxSteps` to distinguish a truncated agent from a well-behaved one.

## Cost and Latency

```typescript
maxCost(0.01)      // fail if a single case costs more than $0.01
maxLatency(3000)   // fail if a case takes more than 3 seconds
```

`maxCost` requires `aiOptions.cost` to be set on `createEval` — otherwise `estimatedCost` is `undefined` and the scorer always passes with a note in `metadata`.

```typescript
createEval({
  name: 'cost-check',
  aiOptions: {
    cost: {
      'claude-haiku-4': { input: 0.25, output: 1.25 },
    },
  },
  scorers: [maxCost(0.005)],
  // ...
})
```

`maxLatency` uses `ai.totalDurationMs` (available when `createEvlogIntegration` is active) and falls back to `ai.msToFinish` for streaming tasks.

## Throughput

```typescript
minThroughput(50) // at least 50 tokens per second (streaming tasks only)
```

Only populated for `streamText` tasks. Silently passes with a metadata note for non-streaming tasks.

## Token Efficiency

```typescript
maxTokensPerStep(2000) // fail if average tokens per step > 2000
```

High tokens-per-step indicates the agent is re-reading large contexts or making inefficient tool calls on each turn.

## Writing Custom AI Scorers

Every scorer has access to `ctx.event` — the full wide event including `ai.*`. Use `defineScorer` from `@evlog/eval` and `getEventAI` from `@evlog/eval/ai-scorers` to access the typed telemetry:

```typescript
import { defineScorer } from '@evlog/eval'
import { getEventAI } from '@evlog/eval/ai-scorers'

// Fail if the agent never used any tools (suspicious for tool-heavy tasks)
const mustUseTools = defineScorer({
  name: 'must-use-tools',
  score: ({ event }) => {
    const ai = getEventAI(event)
    const count = ai?.toolCalls?.length ?? 0
    return { score: count > 0, metadata: { toolCallCount: count } }
  },
})

// Fail if the model produced suspiciously few output tokens (gave up early)
const minOutputTokens = (min: number) => defineScorer({
  name: `min-output-tokens(${min})`,
  score: ({ event }) => {
    const ai = getEventAI(event)
    const tokens = ai?.outputTokens ?? 0
    return { score: tokens >= min, metadata: { outputTokens: tokens, min } }
  },
})
```
