If you ship an npm package that does any meaningful work — DB client, AI tool, payment helper, framework, anything — your users want to see what it's doing. evlog gives you a clean way to instrument library code without making decisions on behalf of your consumers.

Add evlog instrumentation to my published npm library without owning the host app's logger config.

- Add evlog as a peerDependency (and devDependency for tests), never a hard dependency
- NEVER call initLogger from inside the library — the host app owns global config (drain, redact, sampling)
- Create a small createLibLogger(scope) helper using createLogger({ lib: 'my-lib', scope })
- Inside library functions, accumulate context with log.set() and emit one wide event per logical operation
- Use a stable lib + scope namespace so consumers can filter / sample on it
- Accept an optional `logger` parameter in public APIs so consumers can plug in their own

Docs: https://www.evlog.dev/use-cases/libraries

::

## Rules of the road

| Do | Don't |
|----|-------|
| Use `createLogger` (and `log.*` for one-off events) | Call `initLogger` from inside the library |
| Accept an optional logger / drain in your public API | Hardcode a drain (Axiom, Sentry, etc.) |
| Default to `silent: false` if a host has set up evlog | Force `pretty: true` or `JSON` output on consumers |
| Document which event names you emit | Emit hundreds of events per call without a sampling story |
| Make logging optional behind a peer dep on `evlog` | Bundle evlog as a hard dependency — keep it as a peer |

The single biggest mistake library authors make: calling `initLogger` from `index.ts`. That replaces the host app's global config (drain, redaction, sampling) with yours. Don't.

## Minimal pattern

```typescript [src/my-lib/log.ts]
import { createLogger } from 'evlog'
import type { AuditableLogger } from 'evlog'

/**
 * Internal logger factory for this library.
 * Never calls initLogger — the host app owns the global config.
 */
export function createLibLogger(scope: string): AuditableLogger {
  return createLogger({ lib: 'my-lib', scope })
}
```

```typescript [src/my-lib/process.ts]
import { createLibLogger } from './log'

export async function processWebhook(payload: WebhookPayload) {
  const log = createLibLogger('webhook')
  log.set({ type: payload.type, source: payload.source })

  try {
    const result = await handle(payload)
    log.set({ status: 'ok', result })
    return result
  } catch (err) {
    log.error(err as Error)
    throw err
  } finally {
    log.emit()
  }
}
```

If the host app has called `initLogger` and configured a drain, your events flow through it automatically. If they haven't, the events still print to the console (in dev) or get serialized to stdout (in prod) — they're never lost, but they don't go anywhere fancy.

## Accepting a logger as an argument

For libraries called inside an HTTP request, the cleanest shape is to accept the request logger so your fields contribute to the host's wide event:

```typescript [src/my-lib/index.ts]
import type { AuditableLogger } from 'evlog'

interface ProcessOptions {
  logger?: AuditableLogger
}

export async function processOrder(order: Order, options: ProcessOptions = {}) {
  options.logger?.set({ mylib: { orderId: order.id, items: order.items.length } })
  // ... do work ...
}
```

```typescript [server/api/checkout.post.ts (consumer)]
import { useLogger } from 'evlog'
import { processOrder } from 'my-lib'

export default defineEventHandler(async (event) => {
  const log = useLogger(event)
  await processOrder(order, { logger: log })
  // The wide event for /api/checkout now includes the mylib fields.
})
```

This pattern composes: the host's wide event accumulates your library's fields without spawning a separate event per call.

## Peer dependency

In `package.json`:

```json
{
  "peerDependencies": {
    "evlog": "^1"
  },
  "peerDependenciesMeta": {
    "evlog": { "optional": true }
  }
}
```

Mark it optional if you want the library to keep working when the host doesn't depend on evlog. Inside your code, fall back to `console.log` when the import isn't available:

```typescript [src/my-lib/log.ts]
let evlogModule: typeof import('evlog') | undefined
try {
  evlogModule = await import('evlog')
} catch {
  // evlog not installed by the host — use console as a fallback
}

export function createLibLogger(scope: string) {
  if (!evlogModule) {
    return {
      set() {},
      info: (msg: string) => console.log(`[my-lib:${scope}]`, msg),
      warn: (msg: string) => console.warn(`[my-lib:${scope}]`, msg),
      error: (msg: string) => console.error(`[my-lib:${scope}]`, msg),
      emit: () => null,
    }
  }
  return evlogModule.createLogger({ lib: 'my-lib', scope })
}
```

::callout{icon="i-lucide-info" color="info"}
There is no `createLibraryLogger` helper today — `createLogger` plus a "do not call initLogger" comment is the documented workaround. See [Honest gaps](/getting-started/vs-other-loggers#honest-gaps-today).
::

## Documenting your events

Publish the event shape so consumers can build dashboards and alerts. A short README section is enough:

```markdown
## Events emitted

- `lib: 'my-lib', scope: 'webhook'` — once per webhook processed. Fields: `type`, `source`, `status`, `result`.
- `lib: 'my-lib', scope: 'sync'` — once per sync run. Fields: `source`, `target`, `records.found`, `records.synced`.
```
