In a monorepo with multiple apps and shared packages, you want one consistent logging setup across everything: the same drain, the same redaction rules, the same field shapes. The pattern below works in Turborepo, pnpm workspaces, Bun workspaces, and Yarn workspaces.

Set up evlog consistently across my monorepo (Turborepo / pnpm / Bun / Yarn workspaces).

- Create a workspace package @myorg/logging that re-exports evlog plus a setupLogging({ service }) helper
- Centralize the drain pipeline (batch + retry + Axiom/OTLP/Sentry) and shared redaction rules in @myorg/logging
- Define a shared AppEvent type so every app gets typed log.set() with org / user / feature fields
- In each leaf app (apps/api, apps/web, apps/worker), call setupLogging({ service: '<app>' }) once at startup
- In shared library packages, import createLogger only — NEVER call initLogger from libraries
- Wire each app's framework integration on top of setupLogging (Nuxt module, Hono middleware, etc.)

Docs: https://www.evlog.dev/use-cases/monorepos
Configuration: https://www.evlog.dev/core-concepts/configuration

::

## Pattern

1. A small workspace package (`@yourorg/logging`) re-exports `evlog` with shared types and a `setupLogging()` helper.
2. Each leaf app (apps/api, apps/web, apps/worker) calls `setupLogging({ service })` once at startup.
3. Shared library packages (`@yourorg/db`, `@yourorg/billing`) import `createLogger` and never touch `initLogger` (see [Libraries](/use-cases/libraries)).

## The shared package

```typescript [packages/logging/src/index.ts]
import type { LoggerConfig, DrainContext } from 'evlog'
import { initLogger as evlogInit, log, createLogger, createRequestLogger, createError, parseError } from 'evlog'
import { createDrainPipeline } from 'evlog/pipeline'
import { createAxiomDrain } from 'evlog/axiom'

export interface SetupOptions {
  service: string
  environment?: string
  pretty?: boolean
}

/**
 * Shared field shape for every wide event in the monorepo.
 * Add domain-specific fields here so every app gets the same typed access.
 */
export interface AppEvent {
  service: string
  environment: string
  org?: { id: string, plan: 'free' | 'pro' | 'enterprise' }
  user?: { id: string, role: 'admin' | 'member' }
  feature?: string
}

const pipeline = createDrainPipeline<DrainContext>({
  batch: { size: 50, intervalMs: 5000 },
  retry: { maxAttempts: 3 },
})

export function setupLogging(options: SetupOptions): void {
  const config: LoggerConfig = {
    env: {
      service: options.service,
      environment: options.environment ?? process.env.NODE_ENV ?? 'development',
      version: process.env.GIT_SHA?.slice(0, 7),
    },
    pretty: options.pretty ?? process.env.NODE_ENV !== 'production',
    redact: process.env.NODE_ENV === 'production',
    drain: pipeline(createAxiomDrain({
      token: process.env.AXIOM_TOKEN!,
      dataset: 'shared-logs',
    })),
  }
  evlogInit(config)
}

export { log, createLogger, createRequestLogger, createError, parseError }
export type { AppEvent }
```

```json [packages/logging/package.json]
{
  "name": "@yourorg/logging",
  "version": "0.0.0",
  "type": "module",
  "exports": { ".": "./src/index.ts" },
  "dependencies": {
    "evlog": "^1"
  }
}
```

## Each app calls `setupLogging` once

```typescript [apps/api/src/index.ts]
import { setupLogging } from '@yourorg/logging'

setupLogging({ service: 'api' })

// rest of bootstrap...
```

```typescript [apps/web/server.ts]
import { setupLogging } from '@yourorg/logging'

setupLogging({ service: 'web' })

// rest of bootstrap...
```

```typescript [apps/worker/index.ts]
import { setupLogging } from '@yourorg/logging'

setupLogging({ service: 'worker', pretty: false })

// rest of bootstrap...
```

Same drain, same redaction, same enricher pipeline — only `service` differs. Filtering by `service = 'api'` in your dashboard is now meaningful.

## Shared library packages

Library packages inside the monorepo follow the same rules as published libraries — never call `initLogger`, accept an optional logger, or use `createLogger` for self-contained operations.

```typescript [packages/billing/src/charge.ts]
import { createLogger } from '@yourorg/logging'
import type { AppEvent } from '@yourorg/logging'

export async function chargeCustomer(orgId: string, amount: number) {
  const log = createLogger<AppEvent>({
    service: 'billing',
    environment: process.env.NODE_ENV ?? 'development',
    org: { id: orgId, plan: 'pro' },
    feature: 'billing.charge',
  })

  try {
    log.set({ amount, currency: 'USD' })
    const result = await stripe.charge(amount)
    log.set({ status: 'succeeded', chargeId: result.id })
    return result
  } catch (err) {
    log.error(err as Error)
    throw err
  } finally {
    log.emit()
  }
}
```

The `<AppEvent>` type parameter on `createLogger` gives you autocompletion and type-checking for `org`, `user`, `feature`, etc. — see [Typed Fields](/core-concepts/typed-fields) for the full pattern.

## Turborepo `turbo.json`

Make sure the logging package is built before any consumer:

```json [turbo.json]
{
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**"]
    },
    "dev": {
      "dependsOn": ["@yourorg/logging#build"],
      "cache": false,
      "persistent": true
    }
  }
}
```

If you publish `@yourorg/logging` to a private registry instead of using it as a workspace, no special config needed — it's just an npm dependency.

## Why a wrapper, not direct `evlog` imports?

You could have every app import `evlog` directly and call `initLogger` themselves. The wrapper buys you:

- **One place to change drains** — switch from Axiom to OTLP without touching 12 apps.
- **Consistent redaction and sampling** — defaults can't drift between services.
- **Shared event shape** — typed access to `org` / `user` / `feature` without copy-paste in each app.
- **Easier upgrades** — when evlog ships a new feature, you wire it once in `@yourorg/logging`.
