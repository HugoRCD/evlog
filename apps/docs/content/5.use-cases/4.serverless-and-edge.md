Set up evlog in my serverless / edge function (Cloudflare Workers, Vercel Edge, AWS Lambda).

- Install evlog: pnpm add evlog
- For Cloudflare Workers: import initWorkersLogger and createWorkersLogger from 'evlog/workers'; init at module load, create a request logger inside fetch, and call ctx.waitUntil(log.emit()) so the runtime doesn't kill the drain
- For Vercel Edge / generic Edge Runtime: call initLogger() once and createRequestLogger({ method, path, requestId }) per request
- For AWS Lambda: initLogger() at module load (cold start), createLogger({ messageId, ... }) per invocation or per SQS message, log.emit() before returning
- NEVER share a single module-level logger across invocations — runtimes are reused
- Pick an edge-compatible drain (Axiom, OTLP/HTTP, custom HTTP) — no fs / Node-only adapters

Docs: https://www.evlog.dev/use-cases/serverless-and-edge
Workers: https://www.evlog.dev/frameworks/cloudflare-workers
Lambda: https://www.evlog.dev/frameworks/aws-lambda

## Cloudflare Workers

The dedicated `evlog/workers` entry point handles cold-start init and request scoping for you. See the full guide at [Cloudflare Workers](/frameworks/cloudflare-workers) — the snippet below is the minimum.

```typescript [src/worker.ts]
import { initWorkersLogger, createWorkersLogger } from 'evlog/workers'
import { createAxiomDrain } from 'evlog/axiom'

initWorkersLogger({
  env: { service: 'edge-api' },
  drain: createAxiomDrain({
    token: globalThis.AXIOM_TOKEN,
    dataset: 'edge-logs',
  }),
})

export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext) {
    const log = createWorkersLogger(request)

    try {
      const data = await handle(request, env)
      log.set({ status: 200 })
      return Response.json(data)
    } catch (err) {
      log.error(err as Error)
      log.set({ status: 500 })
      return new Response('Internal error', { status: 500 })
    } finally {
      ctx.waitUntil(Promise.resolve(log.emit()))
    }
  },
}
```

::callout{icon="i-lucide-info" color="info"}
Always wrap the `emit()` call in `ctx.waitUntil` so Cloudflare doesn't kill the worker before the drain has flushed.
::

## Vercel Edge Functions (without a framework)

For raw `Edge Runtime` handlers (no Next.js / SvelteKit / Astro on top), use `createRequestLogger`:

```typescript [api/edge-handler.ts]
import { initLogger, createRequestLogger } from 'evlog'
import { createAxiomDrain } from 'evlog/axiom'

initLogger({
  env: { service: 'vercel-edge' },
  drain: createAxiomDrain(),
})

export const config = { runtime: 'edge' }

export default async function handler(request: Request) {
  const log = createRequestLogger({
    method: request.method,
    path: new URL(request.url).pathname,
    requestId: request.headers.get('x-vercel-id') ?? crypto.randomUUID(),
  })

  try {
    const data = await handle(request)
    log.set({ status: 200 })
    return Response.json(data)
  } catch (err) {
    log.error(err as Error)
    log.set({ status: 500 })
    return new Response('Internal error', { status: 500 })
  } finally {
    log.emit()
  }
}
```

If you're on Next.js / Nuxt / SvelteKit / Hono on Vercel, use the framework integration instead — it does this for you.

## AWS Lambda

Lambdas have a cold-start phase you only want to pay for once per container. Initialize the logger module-scoped, create a request logger per invocation, flush before returning.

```typescript [src/handler.ts]
import { initLogger, createRequestLogger } from 'evlog'
import type { DrainContext } from 'evlog'
import { createAxiomDrain } from 'evlog/axiom'
import { createDrainPipeline } from 'evlog/pipeline'

const pipeline = createDrainPipeline<DrainContext>({ batch: { size: 1 } })
const drain = pipeline(createAxiomDrain())

initLogger({
  env: {
    service: 'lambda-api',
    region: process.env.AWS_REGION,
    version: process.env.LAMBDA_FUNCTION_VERSION,
  },
  drain,
})

export async function handler(event: APIGatewayEvent) {
  const log = createRequestLogger({
    method: event.httpMethod,
    path: event.path,
    requestId: event.requestContext.requestId,
  })

  try {
    log.set({ stage: event.requestContext.stage })

    const result = await processRequest(event)
    log.set({ status: 200 })

    return {
      statusCode: 200,
      body: JSON.stringify(result),
    }
  } catch (err) {
    log.error(err as Error)
    log.set({ status: 500 })
    return { statusCode: 500, body: 'Internal error' }
  } finally {
    log.emit()
    await drain.flush()
  }
}
```

::callout{icon="i-lucide-triangle-alert" color="warning"}
Set `batch.size: 1` for Lambda — you can't batch across invocations safely because the container freezes between requests. Each invocation flushes its own event before returning.
::

See the dedicated [AWS Lambda guide](/frameworks/aws-lambda) for SQS, S3, and EventBridge triggers (no `request` shape).

## Why wide events fit edge runtimes especially well

Edge functions run with strict CPU and wall-clock budgets. Emitting one event per request instead of N log lines means:

- **Fewer drain calls** — pipeline batches them, but at the edge each call may go to a different region.
- **Smaller egress** — one JSON object vs. four log lines saves ~75% of bytes shipped to your drain.
- **Easier debugging** — you can't `tail -f` a worker, you query a dashboard. One row per request makes that survivable.

## Drain choice at the edge

Not every drain works at the edge. Quick guide:

| Drain | Cloudflare Workers | Vercel Edge | AWS Lambda |
|-------|:------------------:|:-----------:|:----------:|
| Axiom | Yes | Yes | Yes |
| OTLP (HTTP) | Yes | Yes | Yes |
| HyperDX | Yes | Yes | Yes |
| PostHog | Yes | Yes | Yes |
| Sentry (HTTP) | Yes | Yes | Yes |
| Better Stack | Yes | Yes | Yes |
| Datadog | Yes | Yes | Yes |
| Filesystem (`evlog/fs`) | No | No | Yes (write to `/tmp`) |

Anything that talks plain `fetch` to an HTTPS endpoint works at the edge.
