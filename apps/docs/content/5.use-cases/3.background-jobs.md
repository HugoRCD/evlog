Background jobs are the perfect fit for wide events. Each job invocation is a unit of work with a clear start, end, inputs, and outcome — exactly what `createLogger` was built for. One queryable event per job means retry analysis, SLO tracking, and "why did this fail" debugging become instant filters.

Set up evlog in my queue worker or cron job (BullMQ / Inngest / Trigger.dev / native cron).

- Install evlog: pnpm add evlog
- Call initLogger({ env: { service: 'email-worker' }, drain }) once at worker boot
- In each job handler, call createLogger({ jobId, queue, jobName, attempt, maxAttempts }) at the start
- Use log.set() as the job progresses to capture inputs, intermediate state, and outcome
- On failure, call log.error(err) so the wide event records the cause
- ALWAYS call log.emit() in a finally block, then await drain.flush() on SIGTERM
- Add tail-sampling rules to keep failures and slow jobs even if you sample successful runs

Docs: https://www.evlog.dev/use-cases/background-jobs
Sampling: https://www.evlog.dev/core-concepts/sampling

::

## Pattern

1. `initLogger` once at worker boot.
2. `createLogger({ jobId, queue })` at the start of each job handler.
3. `log.set` as the job progresses.
4. `log.error` on failure, `log.emit` in `finally`.
5. `await drain.flush()` on shutdown signal.

## With BullMQ

```typescript [workers/email-worker.ts]
import type { DrainContext } from 'evlog'
import { initLogger, createLogger, parseError } from 'evlog'
import { createAxiomDrain } from 'evlog/axiom'
import { createDrainPipeline } from 'evlog/pipeline'
import { Worker } from 'bullmq'

const pipeline = createDrainPipeline<DrainContext>({
  batch: { size: 50, intervalMs: 5000 },
  retry: { maxAttempts: 3 },
})
const drain = pipeline(createAxiomDrain())

initLogger({
  env: { service: 'email-worker', environment: process.env.NODE_ENV ?? 'development' },
  drain,
})

const worker = new Worker(
  'emails',
  async (job) => {
    const log = createLogger({
      jobId: job.id,
      queue: job.queueName,
      jobName: job.name,
      attempt: job.attemptsMade + 1,
      maxAttempts: job.opts.attempts ?? 1,
    })

    try {
      log.set({ to: job.data.to, template: job.data.template })

      const sent = await sendEmail(job.data)
      log.set({ providerId: sent.id, status: 'sent' })

      return sent
    } catch (err) {
      log.error(err as Error)
      const { why, fix } = parseError(err)
      log.set({ status: 'failed', failureReason: why, recommendedFix: fix })
      throw err
    } finally {
      log.emit()
    }
  },
)

async function shutdown() {
  await worker.close()
  await drain.flush()
  process.exit(0)
}

process.on('SIGTERM', shutdown)
process.on('SIGINT', shutdown)
```

```json [Sample event]
{
  "level": "info",
  "service": "email-worker",
  "duration": "342ms",
  "jobId": "82",
  "queue": "emails",
  "jobName": "send-receipt",
  "attempt": 1,
  "maxAttempts": 3,
  "to": "user@example.com",
  "template": "receipt",
  "providerId": "msg_abc123",
  "status": "sent"
}
```

Filter `status = 'failed' AND attempt < maxAttempts` to see jobs that recovered on retry. Filter `status = 'failed' AND attempt = maxAttempts` to see real failures that exhausted retries.

## With Inngest / Trigger.dev / step functions

The pattern is the same — wrap each step in its own `createLogger`, emit at the end:

```typescript [inngest/functions.ts]
import { createLogger } from 'evlog'
import { inngest } from './client'

export const onUserSignup = inngest.createFunction(
  { id: 'on-user-signup' },
  { event: 'user.signup' },
  async ({ event, step }) => {
    const root = createLogger({
      fn: 'on-user-signup',
      eventId: event.id,
      userId: event.data.userId,
    })

    try {
      await step.run('send-welcome-email', async () => {
        const log = createLogger({ fn: 'on-user-signup', step: 'send-welcome-email' })
        try {
          await sendWelcome(event.data)
          log.set({ status: 'ok' })
        } catch (err) {
          log.error(err as Error)
          throw err
        } finally {
          log.emit()
        }
      })

      root.set({ status: 'complete' })
    } catch (err) {
      root.error(err as Error)
      throw err
    } finally {
      root.emit()
    }
  },
)
```

## With native crons (`node-cron`, `croner`)

```typescript [crons/cleanup.ts]
import { createLogger } from 'evlog'
import cron from 'node-cron'

cron.schedule('0 */6 * * *', async () => {
  const run = createLogger({
    cron: 'cleanup',
    schedule: '0 */6 * * *',
  })

  try {
    const deleted = await db.deleteExpiredTokens()
    run.set({ deleted, status: 'complete' })
  } catch (err) {
    run.error(err as Error)
    run.set({ status: 'failed' })
  } finally {
    run.emit()
  }
})
```

## Sampling for high-volume queues

A queue running 10k jobs/min doesn't need every successful event in long-term storage, but you always want errors and slow jobs. Use [tail sampling](/core-concepts/sampling#tail-sampling):

```typescript [workers/init.ts]
initLogger({
  env: { service: 'email-worker' },
  sampling: {
    rates: { info: 5 },
    keep: [
      { duration: 5000 },
    ],
  },
  drain,
})
```

This keeps 5% of successful jobs, 100% of jobs slower than 5 seconds, and 100% of errors (defaults). Force-keep specific jobs by setting `_forceKeep: true` in `log.emit({ _forceKeep: true })` — useful when you detect a critical condition mid-job.
