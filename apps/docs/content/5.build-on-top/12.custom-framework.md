When the framework you use doesn't have an `evlog/<framework>` package yet, you build the integration yourself. evlog ships a small toolkit (`createMiddlewareLogger`, `defineFrameworkIntegration`) that handles the request-context plumbing — you only write the framework-specific glue.

Wire evlog into an HTTP framework (or non-HTTP runtime) that doesn't have a built-in integration.

- For HTTP frameworks, use `defineFrameworkIntegration` from `evlog/toolkit` — declare how to extract the request (`getMethod`, `getUrl`, `getHeaders`), how to attach the logger to the framework's context, and how to handle response/errors
- Or use `createMiddlewareLogger` for the imperative path: create the logger at request start, attach it to the request-scoped context, emit on response end
- For non-HTTP runtimes (queue workers, CLI scripts, cron jobs, durable workflows), use `createRequestLogger` from `evlog/toolkit` directly — wrap each unit of work in a logger lifecycle
- Always call enrichers via the runner (`runEnrichers`) before emitting, never bypass them
- Plug the resulting drain into `createDrainPipeline()` for batching + retries — same as built-in integrations

Docs: https://www.evlog.dev/build-on-top/custom-framework

## Non-HTTP runtimes

If you're integrating a non-HTTP runtime (queue worker, CLI driver, cron job, durable execution engine), use `createRequestLogger` from `evlog/toolkit` directly — see the same [Custom integration](/frameworks/custom-integration) page for the lower-level path.
