evlog is designed to be extended on both ends — you can **observe** what flows through (without altering the pipeline), or **plug into** the pipeline itself (sources, enrichers, drains).

This section bundles every extension surface in one place. Each page is short, focused, and copy-paste oriented.

## I want to…

| Question | Answer |
| --- | --- |
| Subscribe in-process from a script or test | [In-process stream](/build-on-top/in-process-stream) |
| See events live while I dev (browser tab, CLI) | [Stream server](/build-on-top/stream-server) |
| Replay or tail historic logs | [FS reader](/build-on-top/fs-reader) |
| Build a small consumer panel / devtool | [Consumer recipes](/build-on-top/consumer-recipes) |
| React to specific events (alert, side-effect, tracing) | [Plugins](/build-on-top/plugins) |
| Tint events with my own context (tenant id, deploy id…) | [Custom enrichers](/build-on-top/custom-enrichers) |
| Decide whether to keep an event based on the request outcome | [Tail sampling](/build-on-top/tail-sampling) |
| Add a service that receives my logs (Axiom, an internal API…) | [Custom drains](/build-on-top/custom-drains) |
| Combine several drains with retry, batching, fanout | [Drain pipeline](/build-on-top/drain-pipeline) |

## A note on serverless

The in-process [stream](/build-on-top/in-process-stream) and the [stream server](/build-on-top/stream-server) work everywhere a Node-like long-lived process runs — local dev, self-hosted servers, containers (Fly, Railway, Coolify), VMs.

They do **not** work on serverless platforms (Vercel Functions, Cloudflare Workers, AWS Lambda) because each invocation is an isolated process. Use a real broker (Redis Streams, NATS, Pub/Sub) for cross-instance fan-out in those environments.

The [FS reader](/build-on-top/fs-reader), [identity headers](/build-on-top/identity-headers), and the rest of the pipeline / drain extension surfaces work everywhere.
